## Executive verdict

The original design got two load-bearing facts right:

1. the npm wrapper discovers eight per-platform optional packages, and a platform
   tarball can be downloaded and inspected without executing Claude Code; and
2. the binary exposes useful static strings but does not provide a complete,
   declarative `settings.json` schema.

The design was **not ready to implement unchanged**. Its source authority, artifact
shape, provenance, version reproducibility, and validation gate all needed material
correction. The working experiment implements the corrected minimum.

## Accuracy findings and decisions

### 1. “Entire configuration surface” was too broad

The four selected dimensions do not include every Claude Code interface. Hooks,
MCP configuration, agents, plugins, commands, policy-helper envelopes, and other
files have separate contracts. The project now says **selected configuration
interfaces** and names the included dimensions.

### 2. Official docs, not binary strings, define the public surface

The binary proves that a token exists in a shipped artifact, but not whether it is
public, experimental, dead, platform-gated, dependency-owned, or internal. Absence
from docs does not prove “internal,” and a prefix allowlist does not prove “public.”

Correct policy:

- official docs are authoritative for public settings, env vars, flags, actions,
  defaults, descriptions, and documented version bounds;
- SchemaStore supplies reusable validation structure and types, with known lag;
- the binary supplies **undocumented candidates** and corroboration only;
- the changelog supplies release hints, not untyped identifiers to gate blindly.

### 3. Keybindings have better first-class sources than the design recorded

The official keybindings page now publishes contexts, action names, default
shortcuts, and a dedicated SchemaStore URL. Static binary extraction is no longer
the best primary source for public keybinding defaults.

### 4. The original all-schema output shape was semantically wrong

JSON Schema validates JSON instances. `settings.json` and `keybindings.json` are
real JSON instances. CLI flags are argv tokens scoped to commands and subcommands;
they need a catalog. An environment may be represented as a JSON string map, but
that projection must be declared. `$ref`-composing all four would validate an
invented aggregate document that Claude Code never reads.

Correct contract:

- `settings.schema.json`
- `env.schema.json` with an explicit environment-object representation
- `keybindings.schema.json`
- `flags.catalog.json`
- `keybinding-defaults.catalog.json`
- `manifest.json` as the heterogeneous index

### 5. A scalar `x-source` is not field-level provenance

Existence, type, description, default, enum, and version bounds can come from
different sources. A Boolean such as `x-corroborated` loses the evidence needed to
audit a conflict. The corrected model records evidence per fact, plus source and
artifact SHA-256 digests. Confidence/status is derived from evidence rather than
stored as an unexplained Boolean.

### 6. Mutable docs prevent historical reproducibility unless captured

`code.claude.com/docs/en/*.md` is a live, mutable endpoint. Filtering current docs
by occasional `min-version`/`max-version` markers cannot reconstruct a complete
historical schema. A release job must therefore record or archive the exact source
bytes/digests it used. A later regeneration for an old version is a **new
observation**, not guaranteed reproduction, unless the original snapshot exists.

The npm version and integrity identify the shipped package. The matching Anthropic
Git tag can pin its changelog, but the public repository does not contain the docs
pages themselves.

### 7. Changelog backticks are not a safe completeness gate

Release notes contain paths, commands, settings, variables, examples, and ordinary
code tokens in backticks. They also describe additions, removals, fixes, and
temporary behavior. Grepping every backtick identifier and requiring it in an
artifact creates false failures and false classifications.

Use a typed, confidence-scored changelog classifier only for hints. A high-confidence
assertion requires the artifact kind and change verb to be explicit. The source
diff itself remains the primary change record.

### 8. The original validation corpus gate was too weak

SchemaStore's settings schema currently uses `additionalProperties: true`, so an
unknown settings key passes validation. A positive corpus with “zero false
negatives” therefore cannot prove catalog completeness and barely tests unknown-key
handling. Count floors catch collapse but not false additions or incorrect types.

A production gate needs:

1. meta-schema compilation and local `$ref` resolution;
2. official and SchemaStore positive **and negative** fixtures;
3. curated boundary and mutation tests for types, enums, aliases, and nesting;
4. source-specific bounded sections and structural canaries;
5. previous-release semantic diff with both drop and growth budgets;
6. manifest/source/artifact digest verification;
7. explicit drift reports for docs-only and SchemaStore-only facts;
8. no publication of binary-only candidates into the public catalog.

Opt-in scrubbed real configs can supplement these tests. Scraping public dotfiles is
not required and should not be the privacy baseline.

### 9. `versions.json` cannot safely map the current version to its own commit SHA

Putting a commit's SHA inside a file in that same commit changes the commit SHA.
Use a tag/ref plus artifact/manifest digests, or generate an external registry after
the commit. The audited design uses version → tag + manifest digest.

### 10. Package acquisition should verify integrity

Deriving the platform matrix from `optionalDependencies` remains correct. A direct
registry tarball fetch is simpler than `npm pack` in automation and avoids lifecycle
concerns; in either case verify `dist.integrity`, reject version mismatches, and
never retain or publish the binary. Static candidate extraction can happen in an
ephemeral directory.

### 11. Do not assume documentation prose is freely redistributable

Names, enum literals, default key combinations, and other interoperability facts are
different from copying hundreds of full documentation descriptions. SchemaStore
declares Apache-2.0 and can be used subject to that license and attribution. The
official docs do not expose an equivalent open-content license in the collected
pages. Until counsel or explicit permission resolves that, public artifacts should
retain identifiers, source links, evidence hashes, and short factual annotations,
not wholesale docs prose. This is a risk-control decision, not legal advice.

## Recommended source policy

| Fact                                          | Primary                       | Secondary                                 | Binary-only handling              |
| --------------------------------------------- | ----------------------------- | ----------------------------------------- | --------------------------------- |
| public setting existence/prose/version bounds | official settings docs        | SchemaStore                               | candidate only                    |
| setting types/constraints                     | SchemaStore                   | official examples; conservative inference | never inferred from a token alone |
| public env existence/prose                    | official env docs             | binary corroboration                      | separate candidate report         |
| public top-level flags/prose                  | official CLI docs             | scoped Commander extraction               | separate candidate report         |
| flag arity/default/choices/command path       | scoped static Commander model | docs examples                             | unknown if not proven             |
| keybindings validation                        | dedicated SchemaStore schema  | official docs                             | candidate only                    |
| public actions/defaults                       | official keybindings docs     | SchemaStore + binary                      | report drift                      |
| release identity/platform matrix              | exact npm metadata            | GitHub tag/release                        | not applicable                    |
| release change hints                          | changelog at matching tag     | artifact diff                             | never a raw-backtick gate         |

## Experiment result

[`experiments/version-1/generate.mjs`](../../experiments/version-1/generate.mjs) fetched the real sources
for `2.1.207`, bounded the intended tables by headings and headers, emitted the six
corrected artifacts, and recorded source/artifact digests. The validation run:

- compiled all three JSON Schemas with Ajv;
- validated real-format settings, environment, and keybindings examples;
- enforced conservative source-specific count floors;
- checked manifest/catalog count agreement; and
- failed if a documented keybinding action was missing from the validator.

Observed drift was retained in `manifest.json`: seven docs-only untyped top-level
settings, one dotted setting needing structural resolution, zero documented actions
missing from the keybindings schema, and 13 schema actions absent from the current
official default tables. Row-level version bounds excluded three retired env rows
and one retired flag row, leaving 287 env properties and 70 current flag records.
Clause-level version markers remain evidence and are not misread as the introduction
version of the whole row. That visible drift is the behavior the production system
should preserve.
